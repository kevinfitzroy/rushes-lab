# 调研:文件管理系统选型

> **调研日期:** 2026-05-15
> **结论摘要:** 推荐 **TrueNAS Scale + ZFS + Nextcloud(主 UI)+ 自研 FastAPI(旁路)**(候选 C)。下一分歧:Nextcloud 是否作为唯一 Web 层(待用户决定)。
> **状态:** v0.1 草稿,主对话推进中

## 1. 已校准的关键约束

| 维度 | 值 | 来源 |
| --- | --- | --- |
| 总数据量(5 年内) | **100 TB** | 用户口述 2026-05-15 |
| 单文件尺寸 | < 1 GB,以 < 500 MB 为主 | 用户口述 |
| 估算文件数量 | **50 万 - 100 万**(100 TB ÷ 100-200 MB 推算) | 派生 |
| 内容形态 | 短视频成片 + 拍摄原片 都入库 | 用户口述 |
| 工作负载 | 非剪辑挂载;走"下载-剪-回传"或 web 浏览 | 用户口述 |
| 部署形态 | 内网为主,异地访问需要 | v2 文档 |
| 商业付费容忍度 | **0(纯开源)** | 用户口述 |
| 审批通道 | 企业微信(已决) | 用户口述 |
| AI 接入 | 阶段一含 自动标签 / 转写 / 向量检索 | v2 文档 |

## 2. 候选裁剪

按上述约束筛选:

| 候选 | 结论 | 主要理由 |
| --- | --- | --- |
| TrueNAS Scale + ZFS + Nextcloud | ✅ 候选 A | 单/双机即覆盖 100 TB;NC 提供成熟 Web/权限/分享/外部存储指向 SMB |
| TrueNAS Scale + ZFS + 自研 FastAPI Web | ✅ 候选 B | UI 完全可控,深度集成顺手;UI 工作量翻倍 |
| **TrueNAS Scale + ZFS + Nextcloud + FastAPI 旁路** | ✅ **候选 C(推荐)** | NC 干通用 UI,FastAPI 干企微/AI/审批代理;v2 实际想走的形态 |
| MinIO 单/双节点 + 自研 Web | 🟡 备选 | 对象存储语义可用;但 Web/ACL/审计全自研,在已有 ZFS+SMB+NC 路径下不增量 |
| Seafile | 🟡 备选 | 加密 + 同步好;视频浏览能力比 NC 弱,生态偏个人云 |
| SeaweedFS / JuiceFS | ❌ 排除 | 为海量小文件设计,百万文件以内属于过度工程 |
| Ceph(RGW / CephFS) | ❌ 排除 | 工程级解决方案,100 TB 用 Ceph 是过度;百人公司运维不健康 |
| 纯 NFS/SMB + 任意前端 | ❌ 排除 | 不是"文件管理系统",只是协议层 |
| 商业 MAM(iconik/CatDV/Mimir) | ❌ 排除 | 违反"纯开源"约束;且短视频场景下 MAM 的代理/时间码核心增值用不上 |

## 3. 推荐方向:候选 C 的分层结构

```
┌────────────────────────────────────────────────────────────┐
│  Web/UX 层                                                  │
│    ├─ Nextcloud(浏览 / 上传 / 分享 / 权限 / 审计 / 桌面客户端) │
│    └─ 自研 FastAPI 页面(审批申请 / AI 检索 / 任务面板)     │
├────────────────────────────────────────────────────────────┤
│  业务服务层(FastAPI + Celery)                              │
│    • 企微 OAuth / 审批对接 / userid 映射                    │
│    • AI 自动标签 / 转写 / 向量索引(旁路,inotify/cron 触发) │
│    • 敏感目录下载代理 + 签名 URL 签发                       │
├────────────────────────────────────────────────────────────┤
│  存储层:TrueNAS Scale + ZFS                                 │
│    • SMB 共享暴露给 Nextcloud 作为外部存储                  │
│    • 同一目录被 FastAPI 后端按 POSIX 路径直读直写           │
│    • ZFS 快照(30/90/365)+ 异地对象存储冷备(OSS/COS)     │
└────────────────────────────────────────────────────────────┘
```

NC 与 FastAPI 通过共用 ZFS 同一目录实现数据同源。NC 写入后由 inotify watcher 或 cron 扫描通知 FastAPI 做缩略图/AI 索引,无需双写。

## 4. 敏感目录下载审批的挂载方式

下游审批接入有三种实现路径,需要在 ADR 中敲定:

| 选项 | 描述 | 工作量 | UX |
| --- | --- | --- | --- |
| (a) Nextcloud Files Access Control + Webhook | NC 拦截下载 → 触发我方 webhook → 我方建企微审批 → 回写 NC 放行/拒绝 | 中(NC App 配置 + Webhook) | 全程在 NC 内 |
| **(b) NC 关闭敏感目录下载权限 + FastAPI 下载代理 + 签名 URL** | 用户在 NC 浏览,点下载跳转到 `/apply-download` → 建企微审批 → 通过后签发临时签名 URL | **低**(自研下载代理) | 有跳转断点 |
| (c) 自定义 Nextcloud PHP App | 写 NC App 拦截下载请求 | 高(NC App PHP 开发栈) | 全程在 NC 内 |

**MVP 推荐 (b)**:实现成本最低,边界清晰(NC 管文件,FastAPI 管审批/业务)。后期如体验断点严重再升级到 (a)。

## 5. 下一分歧点(待用户决定)

候选 A / B / C 的实质性分歧是 **"Nextcloud 是否作为唯一 Web 层"**:

- **要 Nextcloud**(A 或 C):UI 开箱,社区维护,桌面/移动客户端现成;深度集成走 webhook/旁路。
- **不要 Nextcloud**(B):全 UI 自研 FastAPI + Vue/React;深度集成顺手,UI 工作量翻倍。

主对话下一轮收敛此点。

## 7. 排除项归档(为什么不)

> 留档,供将来质疑"为何没选 X"时回看。

- **SeaweedFS / JuiceFS**:为海量小文件 / POSIX-on-对象存储 设计,百万文件以内属于过度工程
- **Ceph**:工程级解决方案,100 TB 规模负担超过百人公司可承受运维
- **MinIO**:可做,但 Web / 权限 / 审计 全自研;在已有 ZFS+SMB+NC 路径下不增量
- **商业 MAM**:违反纯开源约束;且短视频场景下代理/时间码增值用不上
- **Seafile**:Web 端视频浏览能力弱于 NC;Seafile 强项加密同步与本场景错位
