# ADR-0001: 不走"全自研 FastAPI Web UI"作为文件管理主体

- **状态:** accepted
- **日期:** 2026-05-15
- **决策者:** user + material-storage agent
- **关联:**
  - 调研:[`../research/file-management-system.md`](../research/file-management-system.md) §10 决策切换条件(候选 B)
  - storage-protocols-expert 二轮反馈(主对话归档)A-F 实施约束(inotify 路径过滤、审计断点、`oc_filecache` 不可旁路写、容量影响、撞墙顺序、全文搜索边界)

## 背景

方案 v2 与调研 v0.2 都把"候选 B(全自研 FastAPI Web UI,不依赖 NC/Seafile 等开源底座)"列为兜底。candidate scan 阶段把它当 NC 撞墙后的 fallback,预算工作量 6-9 人月。

storage-protocols-expert agent 在二轮反馈里把"通用文件管理 UI 需要照顾的隐性事项"逐条列了:桌面/移动客户端同步协议、版本树、回收站、外链分享、Web 上传/断点续传、权限模型、审计、缩略图浏览、批量操作、搜索 UX、按 mime 分类、用户/组/标签 等。

用户 2026-05-15 主对话明确:

> "全自研肯定不靠谱,文件管理要照顾的东西还很多。"

## 决策

material-storage **不走"全自研 FastAPI Web UI"路线作为文件管理主体**。整个项目必须基于成熟开源文件管理框架作底座(当前候选:Nextcloud / oCIS / Seafile;具体由 Q1 PoC 选定)。

FastAPI 在所有架构中仅承担**业务专属旁路 + 拼接层**职责:

- 飞书 SSO / 审批触发 / 通知
- AI 自动标签 / 转写 / 向量检索
- 敏感目录下载代理 + 签名 URL
- 自研业务页面(申请、AI 检索结果、任务面板)

FastAPI **不**承担:通用文件浏览/上传/分享/权限/审计/桌面同步/缩略图 等 — 这些必须由底座提供。

## 影响

- 候选 B 退出选型,不再作为 NC/oCIS PoC 失败时的兜底
- PoC 阶段任一候选都不达标 → 必须找下一个开源候选(而非回炉自研)
- 团队从一开始就要适应"底座 + 旁路"边界划分,FastAPI 不越界做通用文件管理 UI
- 后续若飞书 / 业务变化引入新功能,默认问"底座能不能加 app/plugin",其次问"FastAPI 旁路是否值得",**最后**才考虑自研 UI 部件

## 备选方案与拒绝理由

### A. 全自研(被拒绝的本 ADR 对象)

- 预算 6-9 人月写完桌面/移动同步、Web UI 全套、分享、权限、审计
- **拒绝**:工作量与团队规模不匹配;且这些是已被开源底座长期解决的能力,自研无差异化价值

### B. 部分自研 + 部分用开源底座(混合)

- 例:用 Seafile 做同步层,用自研做 Web UI
- **拒绝**:模块边界难维护;同步层与 Web 层共享元数据/权限模型,拆开协调成本高于"底座统一"

### C. 接成熟开源底座(本决策方向)

- NC / oCIS / Seafile / 其他;具体在 Q1 PoC 中选定
- **接受**
